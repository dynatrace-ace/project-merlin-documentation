2. Monaco configuration

Step 1: Create your Monaco configuration#

First, all entities that shall be created in Dynatrace have to be defined. These entities will eventually be deployed by Monaco. For the sake of reusablitly, DynaWizard uses the same syntax. Start by creating a folder monaco. This folder has to contain Dynatrace environment configuration (environment.yaml) and at a projects subfolder with at least one project. Please refer to the official Monaco documentation for more details.

caution

As of now, only a single configuration is supported per configuration type. This would NOT work:

config:- sample2: sample2.json- sample3: sample3.json...
caution

There might be dependencies in some fields between configurations, please make sure to check for those before creating/updating the configs.

note

To update a static value in a monaco template: Go into Dynatrace UI and change the options to the desired state. Then, use the download process to get the latest configuration files and grab the new static value from the json file.

Step 2: Parameterize Monaco configuration#

Although any default Monaco configuration could in theory be applied, DynaWizard allows us to parameterize and therefore reuse configuration for multiple applications. Parameters, for which DynaWizard asks for input during an onboarding flow, have to be defined in wizard.json (See Step 3) and mapped in the according Monaco configuration type(s). Any value in a Monaco configuration type configuration yaml can be parameterized. It's as easy as wrapping a parameter id in double curly brackets: "{{ parameter_id }}". For example:

config.yaml
config:- sample1: sample1.jsonsample1:- name: "{{ applicationName }}" // Instead of the name place a variable to use in the Dynatrace Wizard template- filter: "{{ teamName }}" // Another variable
caution

An entity's name is also it's unique identifier. In case a template is re-applied and a name of a configuration-type's entity is changed, a new (duplicate) entity will be created. It is recommended that this behavior is communicated accordingly, for example by adding a note to any parameters that change an entity's name.
